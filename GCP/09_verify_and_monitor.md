# 09 - Verify and Monitor Deployed Services

## Step 1: Get the service URLs

After deploying all four services, retrieve their URLs:

```bash
# Set region variable
REGION="<YOUR_REGION>"

# Get URLs for all four services
SMOLSEARCH_URL=$(gcloud run services describe smolsearch       --region ${REGION} --format "value(status.url)")
RAGIFY_URL=$(gcloud run services describe ragify               --region ${REGION} --format "value(status.url)")
AGENTFLOW_URL=$(gcloud run services describe agentflow         --region ${REGION} --format "value(status.url)")
LLMOPS_URL=$(gcloud run services describe llmops-baseline      --region ${REGION} --format "value(status.url)")

echo "SmolSearch:     $SMOLSEARCH_URL"
echo "RAGify:         $RAGIFY_URL"
echo "AgentFlow:      $AGENTFLOW_URL"
echo "LLMOps:         $LLMOPS_URL"
```

Write these down. You will use them in curl commands throughout this file.

## Step 2: Health check all four services

```bash
# Health check: SmolSearch
# Replace <SMOLSEARCH_URL> with the actual URL if not using shell variables
curl -s "${SMOLSEARCH_URL}/health"
# Expected: {"status": "ok"}

# Health check: RAGify
curl -s "${RAGIFY_URL}/health"
# Expected: {"status": "ok"}

# Health check: AgentFlow
curl -s "${AGENTFLOW_URL}/health"
# Expected: {"status": "ok"}

# Health check: LLMOps Baseline
curl -s "${LLMOPS_URL}/health"
# Expected: {"status": "ok"}
```

The first request after a period of inactivity will be slow (cold start, up to 60 seconds). If `curl` hangs for more than 90 seconds without a response, check the logs (Step 4).

Use `-s` (silent mode) to suppress curl's progress output so you only see the response body. Use `-w "\nHTTP %{http_code}\n"` to also print the HTTP status code:

```bash
curl -s -w "\nHTTP %{http_code}\n" "${SMOLSEARCH_URL}/health"
```

## Step 3: View Cloud Run in the web console

The GCP Console provides a visual overview of all your Cloud Run services.

1. Open `https://console.cloud.google.com`
2. Make sure you are in the correct project (check the project selector in the top bar)
3. Click the navigation menu (three horizontal lines, top left)
4. Scroll to "Serverless" and click "Cloud Run"
5. You will see a list of all services with their region, last deployed time, and URL

Click on any service (e.g., `smolsearch`) to see:
- **Metrics tab**: request count, request latency, instance count over time
- **Revisions tab**: all deployed revisions with their traffic split
- **Logs tab**: log output from the running containers
- **YAML tab**: the full service configuration as YAML

## Step 4: Cloud Logging (Logs Explorer)

Cloud Logging stores all stdout/stderr output from your containers. The Logs Explorer UI lets you query and filter these logs.

### Access Logs Explorer

1. In the GCP Console, navigate to "Logging" > "Logs Explorer"
   Or go directly: `https://console.cloud.google.com/logs/query`

### Filter logs for a specific service

In the query box at the top, enter:

```
resource.type="cloud_run_revision"
AND resource.labels.service_name="smolsearch"
```

Click "Run query". You will see all log entries from the `smolsearch` service.

To filter by severity (only show errors):

```
resource.type="cloud_run_revision"
AND resource.labels.service_name="smolsearch"
AND severity>=ERROR
```

To filter to a specific time range, use the time selector at the top right of the query box.

### Understanding log fields

Each log entry has these fields:

- `timestamp`: when the log was written (UTC)
- `severity`: DEBUG, INFO, WARNING, ERROR, CRITICAL
- `resource.labels.service_name`: the Cloud Run service name
- `resource.labels.revision_name`: the specific revision (e.g., `smolsearch-00003-xyz`)
- `resource.labels.location`: the region
- `textPayload` or `jsonPayload`: the actual log message

For HTTP request logs (automatically generated by Cloud Run, not your app):
- `httpRequest.requestMethod`: GET, POST, etc.
- `httpRequest.requestUrl`: the full URL path
- `httpRequest.status`: HTTP status code (200, 404, 500)
- `httpRequest.latency`: total request duration including cold start
- `httpRequest.responseSize`: response body size in bytes

### Find cold start time in logs

When a container starts from cold, Cloud Run writes a log entry with a message like:
```
Starting container
```

The latency on the next request log entry includes the cold start time. To see exactly how long the cold start took, look at the first request after a container start:

```
resource.type="cloud_run_revision"
AND resource.labels.service_name="smolsearch"
AND httpRequest.requestUrl="/health"
```

Sort by timestamp descending. The first request after a cold start will have a latency of 30--60 seconds. Subsequent requests will be 100--500ms.

## Step 5: Cloud Run metrics

In the Cloud Run service view (not Logs Explorer), click the **Metrics** tab to see:

- **Request count**: total requests per second, broken down by response code
- **Request latency**: p50, p95, p99 latencies over time
- **Instance count**: number of container instances running at each point in time

Instance count is the most useful metric for understanding scale-to-zero behavior. You will see it drop to 0 a few minutes after traffic stops, and spike up when traffic arrives.

## Step 6: Test each service endpoint

These curl commands test the actual API functionality, not just the health endpoint. The first time you run each command after a cold start, expect a 30--60 second delay.

### SmolSearch

```bash
# Index documents
# Replace <SMOLSEARCH_URL> with the actual URL
curl -X POST "${SMOLSEARCH_URL}/index" \
  -H "Content-Type: application/json" \
  -d '{
    "documents": [
      "LangChain is a framework for building LLM-powered applications.",
      "FAISS enables fast approximate nearest-neighbor search over dense vectors.",
      "Sentence transformers encode text into semantic embedding vectors."
    ]
  }'
# Expected: {"indexed": 3}

# Search
curl -X POST "${SMOLSEARCH_URL}/search" \
  -H "Content-Type: application/json" \
  -d '{
    "query": "What is LangChain?",
    "k": 2
  }'
# Expected: {"results": ["LangChain is a framework...", "..."]}
```

### RAGify

```bash
# Index documents for RAGify
curl -X POST "${RAGIFY_URL}/index" \
  -H "Content-Type: application/json" \
  -d '{
    "documents": [
      "The capital of France is Paris.",
      "The Eiffel Tower was built in 1889.",
      "French cuisine is famous worldwide."
    ]
  }'
# Expected: {"indexed": 3}

# Query with RAG
curl -X POST "${RAGIFY_URL}/query" \
  -H "Content-Type: application/json" \
  -d '{
    "question": "When was the Eiffel Tower built?"
  }'
# Expected: {"answer": "The Eiffel Tower was built in 1889.", "sources": [...]}
```

### AgentFlow

```bash
# Run an agent task
curl -X POST "${AGENTFLOW_URL}/run" \
  -H "Content-Type: application/json" \
  -d '{
    "task": "What is 15 plus 27?"
  }'
# Expected: {"result": "42", "steps": [...]}
```

### LLMOps Baseline

```bash
# Query the baseline model
curl -X POST "${LLMOPS_URL}/query" \
  -H "Content-Type: application/json" \
  -d '{
    "prompt": "Explain what a transformer model is in one sentence."
  }'
# Expected: {"response": "...", "latency_ms": 123}

# Check metrics endpoint
curl -s "${LLMOPS_URL}/metrics"
# Expected: {"n_calls": 1, "avg_latency_ms": 123, ...}
```

## Step 7: How to roll back to a previous revision

```bash
# List all revisions for a service
gcloud run revisions list \
  --service smolsearch \
  --region ${REGION}
```

Output:

```
REVISION             ACTIVE  SERVICE     DEPLOYED                 AUTHOR
smolsearch-00005-abc yes     smolsearch  2024-01-16 10:00:00 UTC  github-deployer@...
smolsearch-00004-def           smolsearch  2024-01-15 09:00:00 UTC  github-deployer@...
smolsearch-00003-ghi           smolsearch  2024-01-14 08:00:00 UTC  github-deployer@...
```

`ACTIVE: yes` means 100% of traffic is going to that revision.

To roll back to revision `smolsearch-00004-def`:

```bash
# Send 100% of traffic to the previous revision
# Replace <REVISION_NAME> with the revision name from the list
gcloud run services update-traffic smolsearch \
  --to-revisions=smolsearch-00004-def=100 \
  --region ${REGION}
```

After this command, all traffic goes to the old revision. The newest revision still exists but receives no traffic. To roll forward again, deploy a new image or restore traffic to the latest revision.
